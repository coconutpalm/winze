# CLAUDE.md Integration Guide

This document provides the CLAUDE.md instructions that teach Claude Code
how to use the planning workflow and search tools. Without these
instructions, the AI won't know the system exists.

## Why CLAUDE.md Matters

The planning tool provides *capability* (semantic search over your docs).
CLAUDE.md provides *behavior* (when and how to use it). You need both.

Specifically, CLAUDE.md encodes:

- **Search first**: Before starting any task, search for related prior work.
- **Archive on completion**: Move finished work from `dev/` to `complete/`.
- **Don't read files directly**: Use `search_plans` to find relevant
  context rather than reading files one by one.

## Essential CLAUDE.md Sections

Copy and adapt the sections below into your project's `CLAUDE.md`.
Adjust paths, tool names, and project-specific details to match your setup.

### Section 1: Searching Plans Documents

This section tells the AI what tools are available and how to use them.

```markdown
## Searching Plans Documents

A vector-search MCP server indexes all `Plans/` markdown files with
automatic metadata inference. **Always use it instead of reading plan
files directly.**

### Search Tools
- **`search_plans(query, ...)`** — semantic search. Find relevant
  context before starting any task. Results are scoped to the current
  project by default. Supports metadata filters:
  - `status`: "active", "complete", "deferred"
  - `doc_type`: "context", "plan", "story", "report", "codemap"
  - `group`: work-item group name (e.g. "cache-gap-detect")
  - `since`: ISO date filter (e.g. "2026-03-01")
  - `detail`: "full" (default), "summary" (300 chars), "files" (paths only)
  - `dedupe`: true (default) — one result per file
  - `all_roots`: true — search across all registered projects
- **`related_plans(group)`** — find all documents in a work-item group
  plus cross-references
- **`recent_plans(days, doc_type, status)`** — list recently modified
  documents
- **`list_plans()`** — enumerate all indexed files
- **`plans_status()`** — health check: embedding status, file/chunk
  counts, HNSW index health, watcher status per root
- **`register_plans(name, plans_dir)`** — register a project's Plans/
  directory for indexing
- **`list_plan_roots()`** — show all registered project roots

### Indexing
- A **filesystem watcher** automatically re-indexes files as they are
  created, modified, moved, or deleted. Manual `index_plans` calls are
  only needed for INDEX.md/STATUS.md regeneration.
- **NEVER call `index_plans` with `reset: true` automatically.** A reset
  drops and rebuilds the entire index (~30s). Only do this when the user
  explicitly requests it.
- **Do not manually edit** `Plans/INDEX.md` or `Plans/STATUS.md` — they
  are auto-generated by `index_plans`.

### Usage
- **Do not `Read` entire plan files** to orient yourself. Use
  `search_plans` first; only read a specific file if the search result
  identifies it as the one you need.
- Use `search_plans` with `group` filter to find all docs for a work
  item: `search_plans("context", group="my-feature")`.
```

### Section 2: Prior Work & Context Discovery

This section establishes the "search first" protocol.

```markdown
## Prior Work & Context Discovery

Before starting any task, **search the Plans repository** for related
prior work rather than relying on inline summaries:

1. **`search_plans`** — semantic search with metadata filters.
   Example: `search_plans("authentication refactor")` or
   `search_plans("API rate limiting", status="complete")`.
2. **`related_plans(group)`** — find all docs in a work-item group.
3. **`recent_plans(days=7)`** — see what's changed recently.
```

### Section 3: Task Lifecycle

This section tells the AI how to create and archive planning documents.

```markdown
## Task Lifecycle

When you begin a new task, first read the existing code and documentation.
Create a `TOPIC-CONTEXT.md` file summarizing the information future
sessions need. Create a `TOPIC-PLAN.md` file listing the steps.

Planning documents are organized by lifecycle:
- `Plans/todo/` — Active and pending work
- `Plans/todo/deferred/` — Deferred work
- `Plans/complete/<group>/` — Archived completed work, grouped by feature
- `Plans/reference/` — Reusable procedures

**When a task is completed**, move ALL related files from `Plans/todo/`
to `Plans/complete/<group>/` — creating the group subdirectory and
renaming files to drop the prefix (e.g. `CACHE-GAP-DETECT-CONTEXT.md`
→ `cache-gap-detect/CONTEXT.md`). The filesystem watcher detects the
moves and updates the index automatically.
```

## Slash Command Skills

Seven skill files let users invoke tools directly from the chat prompt.
These are installed globally by `make install-winze`.

| Skill | Description |
|-------|-------------|
| `/help-plans` | List all available plans commands |
| `/search-plans <query>` | Semantic search over Plans/ |
| `/index-plans [reset]` | Reconcile or full reindex |
| `/recent-plans [days]` | List recently modified documents (default 7 days) |
| `/related-plans <group>` | Find all documents in a work-item group |
| `/register-plans [dir]` | Register a project's Plans/ directory |
| `/list-plan-roots` | Show all registered roots and watcher status |

## Auto-Memory Integration

If you use Claude Code's auto-memory system (`.claude/projects/.../memory/`),
add an entry reminding the AI to use `search_plans` for plan lookups:

```markdown
## Tool Usage Patterns
- **Always use `search_plans`** (not Grep/Glob) to find and search
  planning documents.
- The filesystem watcher auto-indexes `Plans/` file changes.
  Call `index_plans` only for INDEX.md/STATUS.md regeneration.
  **Never pass `reset: true` unless the user explicitly requests it.**
```

This reinforces the protocol across conversations, even when the AI
hasn't read CLAUDE.md recently.

## Adapting for Your Project

The examples above use the tool names from this MCP server (`search_plans`,
`index_plans`, etc.). If you rename the tools or the `Plans/` directory,
update the CLAUDE.md instructions to match.

Key things to customize:
- **Directory name**: Replace `Plans/` with your chosen directory.
- **Document types**: If you use different type conventions (e.g. `ADR`
  instead of `CONTEXT`), update the lifecycle section.
- **Metadata filters**: The `status`, `doc_type`, and `group` values
  depend on your directory structure and naming conventions. Update the
  search examples accordingly.
- **Jira integration**: If you don't use Jira, remove references to
  Jira issue files and the `jira` metadata field.
